## Part A - The Rotating-Analyst Problem and its Solution

### 5.1 Restating the danger precisely

Each model runs 24/7. Its context window is finite, so the runtime will eventually **compact,
summarize, or truncate** older context. After that event the model has *forgotten* part of its
own prior reasoning. Left unaddressed, the system degrades into a **rotating analyst**: a
different-feeling decision-maker rotates into the seat every few hours - one that forgets why
positions are open, re-tries setups it already lost on, and drifts toward over-confidence. The
agent-memory literature calls the general phenomenon **agent drift**, and it is a *solvency*
risk here, not a cosmetic one.

### 5.2 The core principle

> **The context window is volatile RAM. Assume it can be wiped between any two cycles.
> Therefore nothing that matters for correctness is allowed to live only in the context window.**

We make the LLM **stateless between cycles by design.** A deterministic orchestrator
*reconstructs* the model's entire relevant world from durable on-disk storage at the start of
every cycle and serves it fresh. Compaction of the in-flight context then becomes a
**non-event**, because there was never anything important in there to lose.

This inverts the usual assumption. We do **not** try to keep the context alive across
compaction. We assume it dies, and we rebuild it from ground truth every time. A rebuild from
disk is identical whether the prior context was pristine or had just been compacted to nothing.

### 5.3 The three memory layers

Synthesized from the strongest patterns in the literature (MemGPT/Letta core memory;
TradingAgents P&L-keyed reflection; FinMem / Stanford Generative Agents retrieval scoring;
Reflexion outcome reflection).

```
┌────────────────────────────────────────────────────────────────────────────┐
│  LAYER 1 - CORE MEMORY BLOCK   (always re-injected, every cycle, verbatim)    │
│  Small, fixed, deterministically generated from disk. The model's "RAM".     │
│  → positions + their original theses/stops/targets/age, account & cash,       │
│    today's risk budget remaining, trading_state, top-N recent lessons,        │
│    the prime directives. Regenerated each cycle; never "remembered".          │
├────────────────────────────────────────────────────────────────────────────┤
│  LAYER 2 - DECISION JOURNAL    (append-only, durable, P&L-keyed)              │
│  The real long-term memory + complete audit log. One record per decision,    │
│  later amended with realized outcome + a Reflexion-style written lesson.      │
│  ACTIVE-TRADE records are NEVER compressed. Closed records may be summarized  │
│  into lessons, but the raw journal is retained forever.                       │
├────────────────────────────────────────────────────────────────────────────┤
│  LAYER 3 - RETRIEVAL          (recency × relevance × importance, top-K)       │
│  For each candidate, pull only the most relevant past records and inject      │
│  them. This is how the analyst "remembers" the last time it saw this ticker   │
│  or this setup_type - without holding it in context.                          │
└────────────────────────────────────────────────────────────────────────────┘
```

### 5.4 Layer 1 - the Core Memory Block (defeats amnesia about *now*)

A compact block, schema [`schemas/core_memory.schema.json`](../schemas/core_memory.schema.json),
**regenerated from disk and prepended to every decision prompt, verbatim, every cycle.** It is
*generated by the orchestrator*, never authored or remembered by the model. Because it is
re-served each cycle, a freshly-compacted model and a long-running model see the *same* current
truth. It contains:

- **Identity & prime directives** (a pointer to / inlined summary of the role's constitution: `ANALYST_CONSTITUTION.md` / `AUDITOR_CONSTITUTION.md`).
- **Account state:** equity, settled cash, unsettled cash, cash-reserve status.
- **Open positions**, each with its **original entry thesis, stop, target, age, and current
  P&L** - so the model can never "wake up" not knowing why a position exists.
- **Today's risk budget:** trades used / remaining, daily drawdown used, `trading_state`.
- **Top-N recent lessons** (from Layer 2 reflections) and any **active "do-not-repeat" flags**
  (e.g. "this setup_type lost 4× this month").

Seed example: [`state/core_memory.seed.json`](../state/core_memory.seed.json).

**Why this kills the worst failure mode:** the single most dangerous amnesia is forgetting an
open position or the day's risk state. Those are *always* in the Core Memory Block, regenerated
from ground-truth files (broker-reconciled positions + the persisted risk state). The model
*cannot* be amnesiac about positions or rules, because they are re-pasted every turn.

### 5.5 Layer 2 - the Decision Journal (defeats repeating past mistakes)

Append-only, one record per decision (and per *non*-decision - rejections are logged too),
schema [`schemas/decision_journal.schema.json`](../schemas/decision_journal.schema.json).
Each record captures: timestamp, symbol, `setup_type`, the full signal snapshot, the bull/bear/
uncertainty arguments, all sub-scores and the final confidence, the risk decision and reason,
and the order IDs. When the trade later closes, the record is **amended** with the realized
return (raw and vs SPY) and a **Reflexion-style written reflection** - a short, honest paragraph
on what the thesis got right and wrong relative to the outcome.

- **Active trades are never compressed.** They are operational ground truth.
- **Closed trades may be summarized** into `lessons` (durable, slow-decay), but the **raw
  append-only journal is retained permanently** for audit and backtest replay.
- Reflections are keyed by `symbol` *and* `setup_type`, enabling the system to compute, e.g.,
  "breakout_with_volume has a 28% hit-rate over the last 18 attempts" - feeding both the Risk
  Auditor's "repeated failed patterns" check and the confidence-calibration discount (§5.10).

Example lines: [`state/decision_journal.example.jsonl`](../state/decision_journal.example.jsonl).

### 5.6 Layer 3 - retrieval scoring (defeats context bloat)

We do **not** dump history into the prompt - that *causes* drift. For each candidate, score
every relevant past journal entry and inject only the **top-K** (`signal_params.yaml:
memory_top_k`). The score is the Generative-Agents / FinMem formula:

```
score(memory) = w_rec · recency + w_rel · relevance + w_imp · importance     (each ∈ [0,1])

  recency   = exp(−Δdays / Q)          # Ebbinghaus decay; Q larger for deeper layers
  relevance = cosine(embed(memory), embed(current_candidate))   # same symbol/setup ranks high
  importance= assigned at WRITE time   # a hit stop / blown thesis scores high; a routine win low
```

Following FinMem's financial retune, route by document type into decay layers: **news →
fast-decay (Q≈14d)**, **trade reflections & fundamentals → slow-decay (Q≈365d)**, so a
"missed-stop" lesson outlives a stale headline. Weights and Q values live in
`signal_params.yaml`.

### 5.7 Identity stability - why the seat stops rotating

The analyst's **identity is not in the context; it is in
[`ANALYST_CONSTITUTION.md`](../ANALYST_CONSTITUTION.md)** (and `AUDITOR_CONSTITUTION.md` for the auditor), a static, version-controlled file re-served
verbatim every cycle. Persona, mandate, tone, and hard rules are therefore *constant by
construction*, regardless of how the underlying context window churns. Even across a crash,
restart, or full compaction, the next cycle reconstitutes the **same** operating identity from
the same file. The occupant of the seat is rebuilt identically each time - so there is no
rotation.

### 5.8 The orchestrator loop makes compaction irrelevant (summary)

```
every cycle:
    state      = load_persisted_risk_state_from_disk()      # ground truth, not memory
    positions  = reconcile(broker.get_positions(), journal) # ground truth, not memory
    core_block = build_core_memory(state, positions, account, recent_lessons)
    memories   = retrieve_top_k(candidate, journal)         # recency×relevance×importance
    prompt     = CONSTITUTION + core_block + memories + current_signals
    proposal   = GLM(prompt)                                 # stateless call; context may be fresh or compacted - identical result
    ... validate → consensus → risk → execute → append_to_journal ...
```

Nothing in the decision depends on what the model "remembers" from a previous cycle. Therefore
**compaction cannot change a decision.** That is the whole solution.

### 5.9 Memory-integrity safeguards (memory is also an attack/corruption surface)

- The journal is **append-only**; records are never edited, only *amended* with outcomes via
  new appended events (event-sourcing style), so history can't be silently rewritten.
- Every record carries a hash/sequence number; a gap or hash break → `DISABLED_REVIEW`.
- The Core Memory Block is **regenerated, never trusted from a cache**; if its sources
  (risk-state file, reconciled positions) are unavailable or inconsistent → the cycle is
  **skipped** (fail-safe), never run on partial truth.
- Retrieved memory is fed to the LLM as **data, clearly delimited as untrusted context**, never
  as instructions (prompt-injection hygiene - news text lives in here too).

## Part B - Consensus Layer (#6)

### 5.10 Deterministic combination of model outputs

The Consensus Layer is **code**, reading weights from
[`config/consensus.yaml`](../config/consensus.yaml). It takes the LLM sub-scores
(`technical`, `market`, `news`, all 0–100) and the Risk Auditor's `risk_score` (0–100, higher =
worse) and produces one confidence by a **fixed, inspectable formula**:

```
support     = w_tech·technical + w_market·market + w_news·news        # weights sum to 1
risk_factor = 1 − (risk_weight · risk_score / 100)                    # risk only ever discounts
confidence  = clamp(support · risk_factor, 0, 100)

HARD VETO:  if risk_score ≥ risk_veto_threshold  → REJECT (no trade, regardless of support)
```

Worked example with the spec's numbers (technical 81, market 75, news 62, risk 32) and default
weights (`w_tech 0.5, w_market 0.3, w_news 0.2, risk_weight 0.7, veto 60`):

```
support     = 0.5·81 + 0.3·75 + 0.2·62 = 75.4
risk_factor = 1 − 0.7·0.32 = 0.776
confidence  = 75.4 · 0.776 ≈ 58.5  → below the 70 floor → NO TRADE
```

This is deliberately conservative: a "moderate" risk score of 32 is enough to keep a
strong-looking technical setup out of the book. The exact weights are **calibration parameters
set during backtesting** (doc 09), not sacred - but the *structure* is fixed: technical/market/
news can only build support; **risk can only subtract; confidence can only gate down.**

### 5.11 Preventing confidence inflation over time (the named requirement)

Multi-agent and judge-style LLM setups are documented to drift toward over-confidence
("certainty distortion" compounding over repeated runs). We block this with five deterministic
guards:

1. **No carry-forward.** Confidence is recomputed from scratch every cycle. A number is never
   inherited from a prior cycle's context - impossible anyway, since the context is treated as
   wiped.
2. **External re-anchoring via realized calibration.** The journal stores stated-vs-realized
   hit-rate per `setup_type`. If a setup's *stated* confidence has been running well above its
   *realized* win-rate, the Consensus Layer applies a **calibration discount** to that setup's
   confidence. The model cannot out-talk its own track record.
3. **Risk is structurally subtractive.** The formula makes it arithmetically impossible for
   optimism to overpower an elevated risk score (and the veto is absolute).
4. **Anonymized adversarial split.** The Risk Auditor argument is generated independently and,
   where a judge step is used, arguments are stripped of "who said what" before scoring, to kill
   positional/identity bias and sycophancy. Genuine bull-vs-bear *diversity* (not turn-order
   theatrics) is what makes the debate worth anything.
5. **Confidence is metadata, capped in effect.** Per `03_risk_engine.md §3.7`, confidence can
   only *lower* exposure; 90+ buys nothing beyond the tier cap. So even if inflation slips
   through, it cannot enlarge a position.

### 5.12 What consensus does NOT do

- It does not place trades (that's Execution, after the Risk Engine).
- It does not size positions (that's the Risk Engine).
- It does not let a high technical score "buy back" an elevated risk score beyond the formula.
- It does not trust any single number the model emitted without the upstream validation
  (schema + feed re-check + signal re-derivation) having already passed.
